# 需求分析辅助技能（Requirement Analysis Helper）

## 定位

流程注入器，不执行需求分析本身。与主需求分析 skill（如 superpower 等）搭配使用，向其注入强制约束。只注入要求，不替代主分析 skill 的流程控制。Plan 的具体格式和存放路径由主分析 skill 决定，本 Skill 仅规定 Plan 必须包含的要点。

| 注入约束 | 作用 |
|---------|------|
| 澄清约束 | 逐个提问澄清歧义，每次只问一个问题 |
| 测试规格约束 | 澄清后、Plan 生成前，完成 BDD + E2E 设计 |
| 粒度约束 | 单 Plan ≤ 2000 行；Plan 需包含 UT/FT 实现要求 |

若本 Skill 约束与主分析 skill 约束冲突，向用户提问由用户裁决，禁止静默覆盖。

## 触发条件

| 场景 | 关键词/信号 | 激活的约束 |
|------|------------|-----------|
| 通用需求分析 | "分析需求"、"制定 Plan" | 澄清 + 测试规格 + 粒度 |
| 新模块开发 | "新增模块"、"实现功能" | 澄清 + 测试规格 + 粒度 |
| 迁移/翻译 | "迁移"、"翻译" | 澄清 + 测试规格 + 粒度 + 迁移模板 |
| 增量/Bug 修复 | "增量"、"修复"、"改进" | 澄清 + 粒度（测试规格按需） |

## §0 冲突检测

若本 Skill 约束与主分析 skill 约束冲突，向用户提问由用户裁决，禁止静默覆盖。

## §1 澄清提问

### 核心规则

生成任何 Plan 前，逐个提问澄清需求歧义和决策点。

| 规则 | 说明 |
|------|------|
| 每次只问一个问题 | 禁止一次性抛出多个问题。一个问题 → 结束响应等待回答 → 再问下一个 |
| 澄清后再进入测试规格 | 所有澄清问题解决前，禁止进入 §2 |

### 提问优先级

| 优先级 | 问题类型 | 示例 |
|--------|---------|------|
| P0 | 需求范围歧义 | "只迁移 src/main 还是包含测试？" |
| P1 | 技术决策点 | "instanceof 动态分派用 enum 还是 Box<dyn Any>？" |
| P2 | 架构方向选择 | "异常继承体系用 enum 合并还是独立 struct？" |
| P3 | 依赖关系不确定 | "是否依赖尚未迁移的其他模块？" |
| P4 | 拆分偏好 | "超大文件（>2000行）接受单 Plan 还是强制拆分？" |

不确定是否需要问——就问。宁可多问一个问题，也不要带着假设生成 Plan。

## §2 测试规格设计（澄清后、Plan 生成前）

**时序强制**：BDD 和 E2E 规格设计发生在澄清完成后、Plan 生成前。测试规格是 Plan 拆分和编码的契约依据。

```
需求输入 → 需求澄清（应用分析类主skill） → [Phase1] BDD → 审核 ✅ → [Phase2] E2E → 确认 ✅ → Plan → 编码(UT/FT/E2E)
```

### 2.1 BDD 功能规格

加载 `harness-bdd-design`，生成 FT 级 BDD Feature 文件。

- 遵循 BRIEF 原则；每个 Scenario 恰好 1 个 When-Then 对
- 人工审核通过后，Feature 文件在编码阶段禁止修改

完成标准：
- [ ] Feature 经人工审核
- [ ] 每条业务规则 ≥ 1 个 Scenario
- [ ] Scenario Outline + Examples 覆盖边界值/等价类

### 2.2 E2E 测试规格

加载 `generate-e2e-test-guide`，生成声明式 E2E YAML，补充 BDD 无法覆盖的场景（进程管理、环境配置、多组件编排）。

完成标准：
- [ ] 每个 scenario 的 `when` 和 `then` 非空完整
- [ ] 人工确认 E2E 覆盖 BDD 的所有业务路径

## §3 Plan 拆分与要点约束

### 3.1 拆分规则

**单 Plan 上限 2000 行**——超过则注意力漂移、易遗漏规格。

| 需求类型 | 拆分主维度 | 估算依据 |
|---------|-----------|---------|
| 重构/迁移类（有参考代码） | 目录 | 参考代码目录行数 |
| 其他类（新开发/增量/修复） | 代码行数预估 | 自下而上块估算 |

通用约束：
- 循环依赖的目录**必须合并**到同一 Plan
- 每个 Plan 必须可独立编译 + 独立运行 UT
- 高内聚优先：同业务场景不拆分；共享类型前置单独立 Plan

### 3.2 单 Plan 模式（未触发拆分条件）

预估代码量 ≤ 2000 行时，生成**单个 Plan**，不生成 Start 文档，不生成 Final Plan。该 Plan 额外的职责：

- **FT 实现**：完成BDD Step Definition 测试代码实现
- **E2E 执行**：执行 §2.2 生成的 E2E YAML

### 3.3 多 Plan 模式（触发拆分条件）

预估代码量 > 2000 行时，拆分为 Start + 子 Plan + Final Plan。具体格式和存放路径由主分析 skill 决定。

#### Start 文档（总览与入口）

多 Plan 需求必须有一个 Start 文档。Start 是所有 Plan 的入口，引导其他 Plan 如何使用：

- **需求总目标**：整个需求要达成什么
- **澄清记录**：§1 澄清问答摘要
- **Plan 依赖顺序**：Plan 间的 DAG 依赖关系
- **Plan 执行顺序（可并发）**：哪些 Plan 可同时执行，哪些必须串行

#### 每个子 Plan 必须包含

- **总目标**：引用 Start 文档，一句话概括整个需求目标
- **当前 Plan 目标**：本 Plan 要完成什么，明确 IN/OUT scope
- **任务清单**：原子任务列表，每个可独立验证
- **涉及目录/模块**：修改范围
- **UT 实现要求**：UT 用例清单 + 断言要求（见 §4）

子 Plan 交付物 = 功能代码 + UT 实现。FT（BDD Step Definition）不在子 Plan 中实现，由 Final Plan 统一实现。

#### Final Plan 必须包含

- **总目标**：引用 Start 文档
- **当前 Plan 目标**：集成验证，不写功能代码
- **四项责任**（严格顺序）：
  1. 集成编译——子 Plan 产物无冲突
  2. FT 实现+执行——实现 BDD Step Definition + 执行全量 Feature
  3. E2E 执行——执行 §2.2 生成的 E2E YAML
  4. 代码检视——结构对齐、占位代码、AI 综合检查

## §4 测试分层与职责

### 多 Plan 模式

| 层级 | 覆盖 | 由谁实现 | 执行时机 |
|------|-----|---------|---------|
| **UT** | 单个函数逻辑 | **子 Plan** | 子 Plan 编码后立即 |
| **FT** | 系统端到端行为（= BDD Scenario） | **Final Plan**（Step Definition） | 所有子 Plan merge 后 |
| **E2E** | 完整环境系统行为 | **Final Plan**（执行 YAML） | FT 通过后 |

子 Plan 写功能代码 + UT；Final Plan 不写功能代码，专注 FT 实现 + E2E 执行 + 代码检视。FT 在所有模块就绪后统一实现，避免子 Plan 阶段因依赖缺失导致 FT 编译失败。

### 单 Plan 模式

UT + FT + E2E + 代码检视全部在同一 Plan 中完成。无需跨 Plan 协调，无依赖等待。

## §5 需求类型模板（按需加载）

§1 澄清环节之后，扫描 `templates/*-template.md` 的「§元信息」章节。命中则加载该模板，按其声明的注入点叠加到对应位置。未命中则仅执行通用流程。多个模板命中时向用户确认选择。

### template.md 固定格式（6 章节）

| # | 章节 | 注入位置 |
|---|------|---------|
| 1 | 元信息 | 模板名、适用场景、激活条件 |
| 2 | 附加流程环节 | §1 澄清之后、§2 测试规格之前 |
| 3 | 附加拆分策略 | §3.1 拆分规则 |
| 4 | 附加输出制品 | §6 制品表 |
| 5 | 附加 Final Plan 准入条件 | §3.2 Final Plan 准入 |
| 6 | 依赖的外部 Skill | 调用阶段与职责 |
